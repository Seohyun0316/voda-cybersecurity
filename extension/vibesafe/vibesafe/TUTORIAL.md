# VibeSafe 4일 개발 튜토리얼 (F1 + F2)

AI 생성 코드의 보안·법적 위험을 실시간 표시하는 VS Code extension을 4일 만에 완성하는 가이드. ML 백엔드는 아직 없으므로 로컬 규칙 엔진으로 실제 동작하게 만들고, 백엔드는 설정 하나로 나중에 연결한다.

## 전체 그림 (5분)

```
문서 열림/수정 → (debounce) → Analyzer.analyze(text)   ← rules ↔ remote 교체 지점
                                    │ AnalysisResult { riskScore, findings[] }
                 ┌──────────┬───────┴──────┬───────────┐
                 진단(밑줄+   사이드 패널      상태바       Quick Fix
                 문제 패널)   (webview)                   (💡 자동수정)
```

| 목업 UI | 담당 파일 | 담당 |
|---|---|---|
| 물결 밑줄 + 하단 경고 패널 | `src/diagnostics.ts` | F1 |
| 오른쪽 위험도 패널 | `src/riskPanel.ts` | F1 |
| 상태바 "VibeSafe: 위험 3" | `src/statusBar.ts` | F1 |
| 탐지 규칙 엔진 | `src/analyzer/ruleEngine.ts` | F2 |
| 💡 자동 수정 | `src/codeActions.ts` | F2 |
| ML 백엔드 클라이언트 + mock 서버 | `src/analyzer/remoteAnalyzer.ts`, `mock-server/server.js` | F2 |
| 조립 | `src/extension.ts` | 같이 |

**핵심 규칙 하나만 기억하기**: F1은 `AnalysisResult`만 보고 그리고, F2는 `AnalysisResult`만 만든다. 이 계약이 `src/analyzer/types.ts`에 있고, ML 백엔드도 같은 JSON만 반환하면 그대로 연결된다.

## Day 1 — 세팅 + 계약 합의 (같이)

**오전: 실행까지**

1. Node 18+, VS Code 설치 확인 → `npm install && npm run compile && npm test` (테스트 7개 통과 확인)
2. VS Code로 폴더 열고 **F5** → 새로 뜬 "개발 호스트" 창에서 `sample/auth.py` 열기
3. 목업처럼 빨간 밑줄, 문제 패널, 사이드바 방패 아이콘, 상태바가 보이면 성공

**오후: 코드 투어 + 계약 확인**

합의할 내용은 **CONTRACT.md에 전부 미리 확정되어 있다.** 둘이 같이 읽고 이의 없으면 그대로 시작하면 된다 (좌표 0-based, enum 값, 점수 계산, 파일 소유권, git 규칙, 완료 기준 등). 그다음 계약의 코드 버전인 `src/analyzer/types.ts`를 읽는다. 외울 것은 두 개:

```ts
interface Finding {   // 위험 1건: ruleId, message, detail, severity(error|warning|info),
                      // category(secret|injection|crypto|cost|other),
                      // line/startCol/endCol(★0-based), legal?, fix?
}
interface AnalysisResult {  // 파일 1개 결과: riskScore(0~100), findings[], engine, ...
}
```

그다음 `package.json`의 `contributes`(사이드바 아이콘·웹뷰·명령·설정 선언부)와 `src/extension.ts`의 `activate()` 흐름을 훑는다. git 저장소 만들고 `main` + 기능 브랜치(`f1/*`, `f2/*`)로 운영. `types.ts` 수정만은 반드시 둘이 합의.

## Day 2 — F1: 화면 표시 / F2: 탐지 엔진

### F1: 진단 + 상태바

`src/diagnostics.ts` — `DiagnosticCollection`에 넣으면 밑줄과 '문제' 패널이 공짜로 생긴다:

```ts
const range = new vscode.Range(f.line, f.startCol, f.line, f.endCol);
const d = new vscode.Diagnostic(range, `${f.message} — ${f.detail}`, severity);
d.source = 'VibeSafe';
collection.set(uri, diagnostics);
```

`src/statusBar.ts` — `$(shield)`는 내장 아이콘, 빨간 배경은 `ThemeColor('statusBarItem.errorBackground')`, 클릭 시 패널 열기는 `item.command`.

확인: `auth.py`에서 밑줄 3곳 + 호버 메시지 + 문제 패널(Ctrl+Shift+M) + 상태바 "위험 N".

### F2: 룰 엔진

`src/analyzer/ruleEngine.ts` — 규칙 = 정규식 + 메타데이터. `RULES` 배열에 추가만 하면 된다:

```ts
{ id: 'hardcoded-password',
  pattern: /(password|pwd|secret)\s*=\s*["'][^"']{4,}["']/gi,
  severity: 'error', category: 'secret',
  message: '하드코딩 비밀번호 — 개인정보보호법 §29', ... }
```

할 일: 기존 규칙 6개(비밀번호, API 키, SQL injection, md5/sha1, eval, debug=True) 이해 → 규칙 1~2개 직접 추가(예: 주민등록번호 패턴 `\d{6}-[1-4]\d{6}`, `verify=False`) → `src/test/ruleEngine.test.ts`에 탐지 케이스와 오탐 없는 케이스 테스트 추가 → `npm test`.

## Day 3 — F1: 사이드 패널 / F2: 자동 수정 + 백엔드 통신 구조

### F1: 사이드 위험 패널 (webview)

`src/riskPanel.ts`. 구조는 3단계: package.json에 선언(이미 됨) → `registerWebviewViewProvider` → HTML 렌더링 + 양방향 메시지.

- extension → webview: 분석 끝날 때마다 `view.webview.html = render()`
- webview → extension: `vscode.postMessage({type:'applyFixes'})` / `{type:'gotoLine', line}`

주의 두 가지: 색은 `var(--vscode-*)` 변수만 사용(테마 자동 대응), 사용자 데이터는 `escapeHtml()` 필수(XSS).

확인: 점수 바 채워짐, 항목 클릭 → 해당 줄 점프, "자동 수정 적용" 버튼 동작.

### F2: Quick Fix + 원격 분석기

`src/codeActions.ts` — `Finding.fix.replacement`가 있으면 💡 메뉴 제공:

```ts
const action = new vscode.CodeAction(f.fix.title, vscode.CodeActionKind.QuickFix);
action.edit = new vscode.WorkspaceEdit();
action.edit.replace(document.uri, range, f.fix.replacement);
```

`src/analyzer/remoteAnalyzer.ts` — 계약 JSON을 POST하고 `AnalysisResult`를 받는 클라이언트(이미 완성). **엔드투엔드 테스트**:

1. `npm run mock-server` (가짜 ML 서버, 포트 8788)
2. 개발 호스트 설정에서 `vibesafe.engine` → `remote`
3. `auth.py` 수정 → `[ML] ...` mock 응답이 UI에 뜨면 통신 구조 완성
4. mock 서버 끄고 수정 → 로컬 규칙으로 자동 폴백되는지 확인

## Day 4 — 조립 + 통합 테스트 + 패키징 (같이)

`src/extension.ts` 흐름을 같이 읽는다: `createAnalyzer()` 팩토리 → `analyzeDocument()`가 4개 UI 일괄 갱신 → 이벤트(`onDidChangeTextDocument`는 debounce 필수, 탭 전환, 설정 변경) → 전부 `context.subscriptions.push()`.

**통합 체크리스트** (F5 개발 호스트에서):

- [ ] `auth.py` 열면 1초 내 밑줄 + 패널 + 상태바 갱신
- [ ] 비밀번호 줄 삭제 → 경고 소멸 + 점수 하락
- [ ] quick fix 적용 → 재분석 후 경고 소멸
- [ ] `.py` / `.js` / `.ts` 동작, `.md`에선 무반응
- [ ] mock 서버 켜고 remote 모드 동작 + 끄면 폴백

**패키징**:

```bash
npm install -g @vscode/vsce
vsce package   # vibesafe-0.1.0.vsix → 확장 탭 'VSIX에서 설치'로 누구나 설치 가능
```

디버깅 팁: extension 중단점은 F5로 그냥 동작, webview는 명령 팔레트 → "Developer: Open Webview Developer Tools", `console.log`는 원래 창 디버그 콘솔.

## ML 백엔드 연결 가이드

백엔드가 준비되면 할 일은 두 가지뿐: 설정 `vibesafe.remoteEndpoint`에 URL 입력 → `vibesafe.engine`을 `remote`로 변경. 인증이 필요하면 `remoteAnalyzer.ts`의 `TODO(백엔드 연결 시)` 두 곳에 토큰 주입.

### API 스펙 (F 파트 확정 v1.0 — api-spec.md가 원본, mock-server/server.js가 살아있는 예시)

```
POST /detect
Request : { "code": "<코드 전문>", "language": "python|javascript", "file_name": "auth.py" }
Response: {
  "risk_score": 87,                      // §1 공식, 0~100
  "findings": [ {
    "rule_id": "A04-798-001", "cwe": "CWE-798", "category": "secret",
    "severity": "high|medium|low",
    "line": 3, "start_col": 0, "end_col": 25,   // ★ 전부 0-based
    "message": "...", "detail": "...", "risk_score": 75,
    "legal": { "law", "article", "description", "liability": 2, "sanction": 1 },
    "fix": { "title", "replacement" }
  } ]
}
```

어댑터(`remoteAnalyzer.ts`의 `mapDetectResponse`)는 snake_case → camelCase 변환과
severity 매핑(high→error 등)만 담당한다. 좌표는 0-based로 통일해 변환이 없다.

검증 포인트: 좌표 0-based 준수, severity/category enum 일치, legal의 liability/sanction 값이
CONTRACT.md §1 표와 일치, 10초 내 응답, 백엔드 다운 시 폴백 동작.

### 데이터 피처 정리 (백엔드/ML 팀과 논의용)

**1) extension이 지금 보내는 것 (v1 최소 입력)**

| 필드 | 설명 | ML 관점 용도 |
|---|---|---|
| `code` | 파일 전체 소스코드 | 모델의 주 입력 |
| `language` | python, javascript | 언어별 모델/토크나이저 분기 |
| ~~`fileName`~~ | (현행 스펙엔 없음) | `.env`, `config.py`, `test_*` 등 파일 성격 힌트 — v2 추가 제안 |

**2) v2에서 요청에 추가하면 좋은 필드** (탐지 정확도를 올리는 컨텍스트 — 추가 시 `types.ts`와 API 계약을 같이 수정)

| 필드 | 설명 | 왜 필요한가 |
|---|---|---|
| `filePath` | 프로젝트 내 상대 경로 | 테스트 코드/샘플 코드 오탐 감소 |
| `dependencies` | requirements.txt, package.json 내용 | 사용 프레임워크 파악 (Django ORM이면 SQL injection 위험도 낮게 등) |
| `framework` | flask, django, express 등 | 프레임워크별 취약 패턴 적용 |
| `changedLines` | 이번에 수정된 줄 범위(diff) | 전체 재분석 대신 증분 분석 → 응답 속도·비용 절감 |
| `projectId` / `sessionId` | 익명 프로젝트 식별자 | 파일 간 문맥 연결(config에서 정의한 키를 다른 파일에서 사용 등) |

**3) ML 모델이 내부에서 뽑을 만한 피처** (백엔드 팀 몫이지만 알아두면 대화가 쉬움)

- 토큰/임베딩: 소스코드 토큰 시퀀스 (CodeBERT류 모델 입력)
- 문자열 리터럴 피처: 길이, 엔트로피, 문자셋, 접두사(sk-, AKIA, ghp_) — 하드코딩 키 탐지의 핵심
- 변수명 피처: password/secret/token/key 등 시맨틱 네이밍 패턴
- AST 피처: "할당문 우변이 문자열 리터럴", "문자열 연결이 execute()로 흘러감" 같은 구조 정보 — 정규식보다 오탐이 훨씬 적음
- taint flow: 사용자 입력(source) → SQL 실행/eval(sink) 도달 여부 — injection 계열의 정석 피처
- import 피처: hashlib, sqlite3, pickle, subprocess 등 위험 라이브러리 사용 여부

**4) 학습 데이터 관점**

- 출력 라벨 스키마는 이미 `Finding`으로 정의돼 있으므로, 지금 룰 엔진의 출력을 **약한 라벨(weak label)** 삼아 초기 학습 데이터를 만들 수 있다 — 룰 엔진을 유지하는 또 하나의 이유
- quick fix 적용/무시 이벤트를 (동의하에) 수집하면 사용자 피드백 라벨로 활용 가능

**5) 보안·법적 주의** — 소스코드 전문이 서버로 전송되므로: HTTPS 필수, 서버 측 코드 저장 여부·보관 기간을 백엔드 팀과 합의, 사내 코드 반출 정책 확인. 코드 저장이 부담이면 `changedLines` + 해시만 보내는 설계도 논의할 것.
